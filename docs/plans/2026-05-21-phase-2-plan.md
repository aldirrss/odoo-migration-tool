## Summary

Phase 2 evolves the migration tool from a single-user local utility into a small multi-user workspace organized around **Projects**. A project owns its source/target profile pairing, configuration, extraction state, and cleaning state. On top of this foundation we add three features that were deferred from Phase 1:

1. Per-project **Configuration tab** (transaction date, date-column fallback, allowed modules)
2. **Auto-discovery** of installed Odoo modules from the source DB
3. **Bulk edit** in the cleaning UI (spreadsheet-style inline edits + a Bulk Operations panel)

Phase 2 is built in four sequential milestones (M1 → M4). Each milestone is independently shippable; the order is chosen to avoid rework caused by retroactively scoping data to projects.

---

## Confirmed Design Decisions

| Decision | Choice | Reason |
|---|---|---|
| Connection profile scope | **Shared globally** (not per-project) | One Odoo DB is often reused across migrations; avoid duplication |
| Auth method | **Local credentials + HttpOnly session cookie** (bcrypt) | Internal tool; NextAuth/OAuth overkill |
| RBAC | **Minimal: `user` vs `admin`** | Admin manages users + sees all projects; users see only their own |
| Auto-cover date fallback | **Chain**: `dateFilterColumn` → `date` → `date_order` → `create_date` → `write_date` | Practical default; can override per-table |
| Data migration from Phase 1 | **Drop and start clean** | No production users yet; cleaner than writing a migration script |
| Built-in vs discovered modules | **Coexist** — built-in (base/accounting/pos) remain curated defaults; discovery adds extras | Curated definitions stay authoritative for common tables |
| Bulk edit UX | **Inline cell edit + Bulk Operations panel** | Inline for typo fixes; panel for "set 500 rows" |
| Build order | **Foundation first** (M1 → M2 → M3 → M4) | Prevents rescoping rework |

## Architecture Changes (Overview)

### New staging-DB tables

```
users                       id, email, password_hash, role ("user"|"admin"), created_at, last_login_at
sessions                    id, user_id, token_hash, expires_at, created_at
projects                    id, owner_id, name, source_profile_id, target_profile_id, created_at, updated_at
project_configs             project_id (PK/FK), transaction_date_from, date_fallback_enabled,
                            date_fallback_chain (jsonb), allowed_modules (jsonb), updated_at
discovered_modules          id, project_id, name, label, installed (bool), discovered_at
discovered_tables           id, project_id, module_id (FK), table_name, odoo_model, type ("master"|"transaction"),
                            date_filter_column, import_order, columns (jsonb), confidence ("high"|"medium"|"low"),
                            user_classified (bool)
discovered_relations        id, project_id, from_table, from_column, to_table, to_column, on_delete, source ("introspect"|"manual")
```

### Tables modified

- `connection_profiles` — **new** table that replaces `config/connections.json`. Same schema (id, name, host, port, database, user, encrypted_password, ssl) but as a Drizzle-managed table. Profile passwords keep AES-256-GCM via `ENCRYPTION_KEY`.
- `extraction_jobs` → add `project_id` FK.
- `import_jobs` → add `project_id` FK (denormalized for query convenience; can derive from extraction_jobs).
- `staged_records` → unchanged; scoped to project via `extraction_jobs.project_id`.

### Key code shifts

- **Module registry becomes context-aware.** `getAllTables(projectId)` reads built-in `moduleRegistry` **filtered by** `project_configs.allowed_modules`, then **merged with** rows from `discovered_tables` for that project. All call sites in `lib/migration/*` must accept `projectId`.
- **Env-based config is now a fallback.** `TRANSACTION_DATE_FROM` env var becomes the seed/default for new projects only; per-project `project_configs.transaction_date_from` wins at runtime.
- **All API routes require auth.** Wrap `app/api/**` with a session-checking helper that reads the cookie, loads `users` + scopes queries to the current user's projects (or all projects for admins).

## Milestones

### M1 — Project + Auth Foundation

**Goal:** Everything that exists today still works, but scoped under projects and behind login. No new features.

**Deliverables**
- Drizzle schema additions: `users`, `sessions`, `projects`, `connection_profiles` (replaces JSON file).
- Auth flow: `/login`, `/register` (admin-gated after first user), `/logout`. Bcrypt for password hash, HttpOnly + Secure cookies, CSRF token on mutating routes.
- Middleware: `app/middleware.ts` redirects unauthenticated requests; API helper `requireSession(req)`.
- Project CRUD UI: `/projects` (list) + `/projects/[id]` (detail with sub-routes for connections/extract/staging/validate/import).
- Migration script (one-shot, in `scripts/`): drop legacy `config/connections.json`, reset staging schema. Document this clearly in README — Phase 1 users will lose state.
- Existing pipeline modules (`extractor`, `cleaner`, `validator`, `importer`) accept and persist `projectId`.

**Schema migration**
- `drizzle-kit generate` after editing [lib/db/staging-schema.ts](../../lib/db/staging-schema.ts).
- Manual SQL to drop existing data is fine — no `down` migration needed.

**Acceptance**
- A fresh install: register first user → auto-promoted to admin → create project → add source/target profile → run full extract → see records in `/projects/<id>/staging`.
- Logging out hides projects belonging to other users.

### M2 — Configuration Tab

**Goal:** Each project has a Configuration page where transaction date, date fallback, and allowed modules can be edited without redeploying.

**Deliverables**
- New page: `/projects/[id]/configuration`. Three sections:
  1. **Date**: input for `transactionDateFrom` (default seeded from env).
  2. **Date Fallback**: toggle "Auto-cover date" + editable fallback chain (drag-reorder or text list).
  3. **Allowed Modules**: checkbox list of all modules from `moduleRegistry`. Stored as `string[]` in `project_configs.allowed_modules`.
- API: `GET/PUT /api/projects/[id]/config`.
- `lib/odoo/modules/index.ts` gains `getAllTables(projectId)` and `getAllRelations(projectId)` that consult `project_configs`. Old signatures kept as deprecated for one milestone, then removed.
- Extractor reads `transactionDateFrom` per project (no longer global env at runtime).
- Extractor consults date-fallback chain: if `dateFilterColumn` doesn't exist on the source table (introspected via `information_schema.columns`), iterate the chain. If none match → log warning and skip filter (configurable per-project: skip-filter vs skip-table).

**Acceptance**
- Changing "Allowed modules" on Project A and re-extracting produces a different set of tables than Project B with the same source DB.
- Setting `transactionDateFrom = 2025-01-01` on a project extracts more rows than the default 2026-01-01.
- Forcing extraction on a table whose `dateFilterColumn` doesn't exist now uses the fallback chain instead of erroring.

### M3 — Auto-Discovery

**Goal:** Each project can scan its source DB and propose additional modules/tables/relations beyond the built-in registry.

**Deliverables**
- New page: `/projects/[id]/discovery`. Trigger "Scan source DB" button → background job populates `discovered_modules`, `discovered_tables`, `discovered_relations`.
- Discovery logic in `lib/migration/discovery.ts`:
  - Query `ir_module_module WHERE state='installed'` to get installed Odoo modules.
  - For each module, query `ir_model` joined via `ir_model_data` to get models, then derive PostgreSQL table names.
  - Classify each table as `master` vs `transaction` using a heuristic: presence of any column in {`date`, `date_order`, `invoice_date`, `picking_date`} → transaction; else master. Record `confidence` and let the user override.
  - Introspect `information_schema.table_constraints` + `key_column_usage` for FK relations.
- Discovery results show in a review UI with per-table dropdowns for type, date column, import order, on-delete. User confirms → flagged `user_classified = true`.
- Confirmed discoveries flow into the Configuration "Allowed modules" list (M2) automatically.
- **Built-in registry remains authoritative for any table it already defines** — discovery skips tables that exist in `moduleRegistry`.

**Acceptance**
- Running discovery against a source DB with `mrp` (Manufacturing) installed surfaces `mrp_production`, `mrp_workorder`, etc. with reasonable type guesses.
- User can toggle a discovered module on in Configuration and the next extraction pulls its tables.
- A discovered FK is enforced during the Validate step.

### M4 — Bulk Edit

**Goal:** Cleaning UI scales beyond one-record-at-a-time editing.

**Deliverables**
- Refactor `/projects/[id]/staging/[table]` to use TanStack Table with:
  - Inline editable cells (click cell → input → blur to save). On save: update `stagedData`, flip `isDirty`. Optimistic UI + revert on error.
  - Row-level multi-select checkbox + "select all matching filter".
  - Sticky toolbar with **Bulk Operations** panel:
    - **Set field**: pick column, enter value, apply to selection.
    - **Find & Replace**: regex or literal, scoped to one column or all text columns.
    - **Clear field** (set null).
    - **Revert to source** (copy `sourceData` → `stagedData`, unset `isDirty`).
    - **Soft delete** (set `isDeleted` on selection; runs the existing dependency-check per row, blocks if any `onDelete: block` relation has active dependents).
- API: `PATCH /api/projects/[id]/staging/[table]/bulk` accepts `{ recordIds[], operation, params }` and processes server-side in a single transaction.
- Diff badge per row (count of changed fields vs source); existing split-view editor remains accessible via "Inspect" action for detailed per-record work.
- Pagination + server-side search (`stagedData @> jsonb_path` or `ILIKE` per known text column).

**Acceptance**
- Selecting 500 `res.partner` rows and applying `Set country_id = 105` updates all rows in one request and flips `isDirty` on each.
- A failed bulk delete (FK block on one row) reports which rows failed and leaves the rest untouched.
- Inline editing a single cell still works after the refactor.

## Cross-cutting Concerns

- **Permission checks**: every API handler must call `requireSession(req)` and verify `project.owner_id === user.id` (or `user.role === 'admin'`). Centralize in `lib/auth/guards.ts`.
- **Encryption key rotation**: out of scope for Phase 2. Document that rotating `ENCRYPTION_KEY` still invalidates all stored profiles (now in DB instead of JSON).
- **Background jobs**: extraction, import, and discovery are long-running. Phase 1 polls; keep that. Don't introduce queues yet.
- **Tests**: still none. Phase 2 does not change this — leave for a later phase. Manual verification per acceptance criteria.

---

## Risks

| Risk | Mitigation |
|---|---|
| Phase 1 users lose data when we drop the JSON profile store | Document loudly in README; the tool has no production deployments yet |
| Discovery misclassifies `master` vs `transaction` | Require user confirmation; show `confidence` in UI; never auto-extract unconfirmed tables |
| Bulk operations table-locking large `staged_records` | Process in chunks (e.g. 500 rows per transaction); show progress |
| Multi-user race on the same project (two users editing concurrently) | Out of scope; last-write-wins. Document the limitation |
| Session cookie + CSRF correctness | Use a well-tested helper (e.g. `iron-session` or hand-rolled HMAC tokens); keep auth code small and reviewable |

## Future Work — Notes for Next Iteration

- After M3 lands, consider letting users **save discovered classifications as exportable module definitions** (`.ts` files) so reusable modules can graduate into the built-in registry.
- Bulk edit operations are a natural place to layer **undo** later — keep each operation's diff small enough to invert.
- If multi-user collaboration becomes a real need, add per-project membership + a lightweight presence indicator before reaching for real-time sync.

### Known Limitations carried to Phase 3+

- Extractor not yet consuming user-confirmed `discoveredTables` (helper exists; needs wiring + UI confirmation that an enabled discovered module activates extraction).
- No admin UI for user management.
- No CSRF token on mutating routes (relies on SameSite=Lax cookie).
- No background-job queue; long-running scans hold the request open up to 120s (`maxDuration = 120` set on the scan route).
- Bulk edit per-row failure reporting lists reasons but doesn't yet support partial-undo. The whole-chunk transaction rollback means a single chunk failure marks ALL records in that chunk as failed.
