## Summary

Phase 3 adds three large feature areas on top of the Phase 2 multi-tenant foundation:

1. **Curated module expansion** — extend the built-in `moduleRegistry` from 3 modules (base, accounting, pos) to ~10 modules covering the full Odoo standard footprint relevant for migration: Sale, Purchase, Stock, HR, MRP, CRM, Project, Mail, plus existing base/accounting/pos. Each module ships with proper `importOrder`, relations, and `dateFilterColumn`.
2. **Prefix-based auto-classification** — during `discovery` and during `extraction` of unknown tables, the tool infers module membership from table-name prefix (`sale_*` → sale, `purchase_*` → purchase, `account_*` → accounting, …) and infers master-vs-transaction from column signatures (presence of `date_order`/`date`/`state` + outgoing FKs to masters).
3. **Auto data quality screening** — a post-extraction job walks every staged record and runs a battery of rules (orphan FK, missing required field, malformed translation, duplicate natural key, future/stale dates, suspicious numerics, encoding issues). Findings are persisted into a new `quality_flags` column on `staged_records` (severity + reason codes + per-column detail), surfaced in the staging UI as a colored badge column with a click-to-open popover, and (for `block` severity by default) prevent import until the user acts on them.

## Confirmed Design Decisions

| Decision | Choice | Reason |
|---|---|---|
| Module expansion strategy | **Curated definitions** for the 7 new modules (sale, purchase, stock, hr, mrp, crm, project, mail). Auto-discovery still covers anything custom. | Curated catches the 95% common case with correct `importOrder` + relations; discovery covers the long tail without hardcoding. |
| Screening trigger | **Automatic after extraction completes**, plus a manual "Re-scan" button (per-table or whole-project) | Users want flags ready when they open staging; manual re-scan is needed after bulk edits change row contents. |
| Flag UI | **Dedicated "Flags" column with colored badges + click popover listing every issue (rule code, column, message)** | Most scannable; also makes the row width-stable. |
| Severity policy | **Configurable per-rule in Configuration tab. Default seeds: `orphan_fk` = block, `missing_required` = block, `malformed_translation` = block, `duplicate_natural_key` = warn, `future_date` = warn, `stale_date` = warn, `suspicious_negative` = warn, `encoding_issue` = warn** | Blocks are reserved for things that will cause a hard FK/NOT-NULL/JSONB failure at import time. Warnings flag legitimate-but-unusual data that the user should glance at. |
| Storage | **`staged_records.quality_flags JSONB` + scalar `quality_severity` (`ok|warn|block`) for cheap filtering/index** | One row per record stays the source of truth; no separate `staging_record_flags` join table needed. |
| Scan scope | **All non-deleted staged records in the active extraction job** | Soft-deleted rows are skipped (they will not be imported); re-extraction creates a new job so scan re-runs naturally. |

## Feature 2 — Prefix-based auto-classification

### Prefix → module map

```ts
// lib/odoo/prefix-mapping.ts
export const PREFIX_TO_MODULE: Array<[RegExp, string]> = [
  [/^sale_/,      "sale"],
  [/^purchase_/,  "purchase"],
  [/^pos_/,       "pos"],
  [/^account_/,   "accounting"],
  [/^stock_/,     "stock"],
  [/^mrp_/,       "mrp"],
  [/^hr_/,        "hr"],
  [/^crm_/,       "crm"],
  [/^project_/,   "project"],
  [/^mail_/,      "mail"],
  // base catch-all
  [/^res_/,       "base"],
  [/^ir_/,        "base"],
  [/^uom_/,       "base"],
];
```

### Master vs Transaction heuristic

```ts
function inferTableType(columns: ColumnInfo[]): "master" | "transaction" {
  const hasStateOrDateOrder = columns.some(c =>
    c.name === "state" || c.name === "date_order" || c.name === "date_done"
  );
  const fkColumnCount = columns.filter(c => /_id$/.test(c.name)).length;
  // Heuristic: rich state machine + many FKs → transaction
  if (hasStateOrDateOrder && fkColumnCount >= 3) return "transaction";
  return "master";
}
```

### Integration points

- **discovery.ts** — on every persisted `discoveredTables` row, set `moduleSlug = inferModuleSlug(tableName)` and `tableType = inferTableType(columns)`. User can override in the Discovery UI before confirming.
- **extractor.ts** — only ever extracts tables from `getAllTables(projectId)`, which already merges `moduleRegistry` + `discoveredTables`. So as long as discovery tags them, they flow through naturally.

## Feature 3 — Auto data quality screening

### Rule catalog

| Rule code | Default severity | What it checks | Cost |
|---|---|---|---|
| `orphan_fk` | block | For every `*_id` column on the record, check that the referenced row exists either in non-deleted staging or in target. Uses `getOutgoingRelations()` + `inferFkRelation()`. | High (one lookup per FK; batched) |
| `missing_required` | block | For columns Odoo marks NOT NULL (queried from `pg_catalog` once per table), value in `stagedData` must not be NULL / empty. | Low (column metadata cached) |
| `malformed_translation` | block | If column is a JSONB translation field (per `isTranslationDict`), validate keys match locale regex and values are strings. | Low |
| `duplicate_natural_key` | warn | Configurable natural-key columns per table (e.g. `res_partner.email`, `product_product.default_code`). Flags rows whose value collides with another staged row. | Medium (one GROUP BY per table) |
| `future_date` | warn | Any date/datetime column > today. | Low |
| `stale_date` | warn | Any date/datetime column older than `TRANSACTION_DATE_FROM - 5y`. Often indicates broken legacy data. | Low |
| `suspicious_negative` | warn | Amount columns (`amount_*`, `price_*`, `qty_*`) with value < 0 where business rules say ≥ 0. Per-column allowlist. | Low |
| `encoding_issue` | warn | Text columns containing non-printable ASCII / U+FFFD replacement chars / mojibake patterns. | Low |

### Scan algorithm

```ts
async function runQualityScan(jobId: number, opts: { tableName?: string }) {
  // 1) Load project config → which rules enabled + severity overrides.
  // 2) For each table in scope:
  //    a) Load NOT-NULL column set from pg_catalog (target DB).
  //    b) Load relations from registry + heuristics.
  //    c) Stream staged rows in chunks of 1000 (keyset on id).
  //    d) For each row: run every enabled rule, collect findings.
  //    e) Cross-row rules (duplicate_natural_key): GROUP BY column value.
  //    f) UPDATE staged_records SET qualityFlags, qualitySeverity, qualityScannedAt.
  // 3) Write summary into extraction_jobs (counts by severity).
}
```

Runs **after** extraction's main loop finishes. Cancellable (checks `extraction_jobs.cancel_requested` between chunks). Reported as a sub-step on the extract page ("Scanning quality…").

### `qualityFlags` shape on a row

```json
[
  {
    "rule": "orphan_fk",
    "severity": "block",
    "column": "partner_id",
    "message": "partner_id = 1428 references res_partner.id but no such row exists in staging or target"
  },
  {
    "rule": "future_date",
    "severity": "warn",
    "column": "date_order",
    "message": "date_order = 2099-01-01 is in the future"
  }
]
```

`qualitySeverity` is the worst of the findings: `block` > `warn` > `ok` (no findings).

### UI — `<QualityFlagCell />`

Rendered in a new column at the right edge of the staging table:

- `qualitySeverity = block`: red badge `● 2` (count of findings)
- `qualitySeverity = warn`: yellow badge `● 1`
- `qualitySeverity = ok`: muted green check `✓`
- Click → Radix Popover listing findings: `[badge] column — message`
- Filter toolbar adds dropdown: `All | Block only | Warn only | Clean only`

### UI — Configuration tab: Quality Rules panel

Card with one row per rule:

```
[✓ enabled]  orphan_fk             severity: ( block ▾ )   "FK references row that doesn't exist"
[✓ enabled]  missing_required      severity: ( block ▾ )   "NOT NULL column is empty"
...
```

Save persists into `project_configs.qualityRules`.

### Importer integration

Before importing each row:

```ts
if (record.qualitySeverity === "block" && !record.qualityOverridden) {
  skip(record, reason: "blocked by quality rule");
}
```

`qualityOverridden boolean` — additional column. Set via "Acknowledge & allow import" button in the popover. Audit trail: who/when.
