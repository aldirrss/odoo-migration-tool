# Phase 4 — Custom Module Extraction (Wire Discovery into the Pipeline)

**Date:** 2026-05-21
**Status:** Planning (parked — not yet started)
**Predecessor:** [2026-05-21-phase-3-plan.md](./2026-05-21-phase-3-plan.md) (Phase 3 — Built & Delivered)

---

## Problem

Currently the extractor only walks tables from the **curated `moduleRegistry`** (11 modules: base, accounting, pos, sale, purchase, stock, hr, mrp, crm, project, mail). Custom Odoo modules (e.g. `lm_*`, `mzt_*`, `dtg_*`) are correctly **discovered** and tagged in `discoveredTables` (Phase 2 + Phase 3 M2), but they are **never extracted** because `getAllTables(projectId)` in `lib/odoo/modules/project-scope.ts` ignores discovery data.

This means the auto-discovery feature exists end-to-end on the UI side, but the pipeline silently drops custom-module tables once the user hits Start Extraction.

## Goal

Extend the pipeline so any `discoveredTables` row that the user has confirmed (`userClassified = true`) is treated as a first-class table during extract / validate / import — without duplicating tables that already exist in the curated registry.

## Scope

### In scope
- Merge curated registry + confirmed discovered tables in `getAllTables(projectId)`.
- Merge curated relations + confirmed discovered relations in `getAllRelations(projectId)`.
- Synthesize a `TableDefinition` from a `discoveredTables` row:
  - `tableName` from the row
  - `type` from `tableType` (the Phase 3 M2 column; `master | transaction`)
  - `importOrder` default — propose a custom range `700–799` so customs slot in after the curated modules
  - `dateFilterColumn` inferred from the column set (use the same fallback chain as Phase 2: `date → date_order → create_date → write_date`)
- Synthesize `RelationDefinition` from `discoveredRelations`:
  - `onDelete` defaults to `nullify` (safest for inferred FKs; matches `fk-heuristics.ts`)
  - User can override later via a discovery-edit UI (out of scope)
- Skip discovered tables whose name collides with a curated registry table (curated wins).
- Respect `allowedModules` config: a discovered table is included only if its `moduleSlug` is in the allowed list (or `unknown` → include if a special "Include unclassified" project toggle is on; default off to avoid accidents).

### Out of scope (Phase 5+ candidates)
- Editing `importOrder` per discovered table in the UI.
- Editing `onDelete` per discovered relation in the UI.
- Auto-detecting natural keys for `duplicate_natural_key` quality rule on custom tables.
- Re-discovery on schema drift detection.
- Per-table `dateFilterColumn` override UI for discovered tables.

## Architecture changes

### `lib/odoo/modules/project-scope.ts`

```ts
export async function getAllTables(projectId?: number): Promise<TableDefinition[]> {
  if (projectId == null) {
    return moduleRegistry.flatMap((m) => m.tables)
      .sort((a, b) => (a.importOrder ?? 100) - (b.importOrder ?? 100));
  }

  const allowed = await getAllowedModuleNames(projectId);   // existing
  const curated = filterModules(allowed).flatMap((m) => m.tables);
  const curatedNames = new Set(curated.map((t) => t.tableName));

  const discovered = await loadConfirmedDiscoveredTables(projectId, allowed);
  const synthesized: TableDefinition[] = discovered
    .filter((d) => !curatedNames.has(d.tableName))           // curated wins
    .map(synthesizeDiscoveredTable);

  return [...curated, ...synthesized]
    .sort((a, b) => (a.importOrder ?? 100) - (b.importOrder ?? 100));
}
```

`loadConfirmedDiscoveredTables(projectId, allowed)`:
```sql
SELECT dt.*
FROM discovered_tables dt
WHERE dt.project_id = $1
  AND dt.user_classified = true
  AND (dt.module_slug IS NULL OR dt.module_slug = ANY($2))
```

`synthesizeDiscoveredTable(row) → TableDefinition`:
```ts
{
  tableName: row.tableName,
  type: row.tableType ?? "master",
  importOrder: 700 + hashToBucket(row.tableName, 99),   // stable, deterministic
  dateFilterColumn: pickDateColumn(row.columns),         // null if none
  // No columns metadata declared — extractor uses information_schema fallback
}
```

### `lib/odoo/modules/project-scope.ts` — `getAllRelations`

Same merge pattern: `moduleRegistry` relations + `discoveredRelations` rows (filtered by source table being in the allowed table set).

`onDelete` default = `"nullify"` until user-edit UI lands.

### `lib/migration/extractor.ts`

No code changes needed — already consumes `getAllTables(projectId)` polymorphically. The extractor will run COUNT + keyset over discovered tables exactly like curated ones. Phase 3 M3 quality scan already runs on every staged record regardless of source module, so quality flags will populate for custom modules too.

### `lib/migration/cleaner.ts` / `validator.ts` / `importer.ts`

Validator already walks `getOutgoingRelations(table, projectId)`. As long as that helper merges discovered relations, validator picks them up automatically. Importer iterates `getAllTables(projectId)` → also automatic.

## Edge cases

- **Discovered table with no `id` column.** Extractor keyset assumes `id`. Need to either skip such tables (likely) or extend the extractor to handle composite/no-id tables. Phase 4 decision: **skip + emit a warning row in `table_extraction_status`** with `errorMessage = "table has no integer 'id' column; skipped"`. Out-of-scope to support composite keys.
- **importOrder collisions with `mail` (900+).** 700–799 leaves headroom; mail is 900+. Safe.
- **`moduleSlug` is `unknown` / NULL.** Default: exclude unless project config opts in. Add `projectConfigs.includeUnclassifiedDiscovered boolean default false`.
- **User toggles a discovered row to `userClassified = false` after extract.** Next extract will skip it; previously extracted staged_records remain (until job is reset).
- **Relation target table is itself unrecognized.** Allow — the orphan_fk quality rule will catch it as a finding rather than failing extraction.

## Schema changes

Probably one new column:
```sql
ALTER TABLE project_configs
  ADD COLUMN include_unclassified_discovered boolean NOT NULL DEFAULT false;
```

No other schema changes needed — `discoveredTables` and `discoveredRelations` already carry everything we need from Phase 2 + Phase 3 M2.

## Risks

- **Performance.** A project with 200+ custom tables could explode extraction time. Mitigation: existing keyset pagination + cancel checkpoints already make this bounded.
- **importOrder collisions.** If a custom table FK-depends on another custom table, the deterministic hash buckets may order them wrong. Workaround: relations help validator catch this; importer will skip the dependent row with a clear "parent not yet imported" message. A future iteration could topologically sort discovered tables via discovered relations.
- **Quality scan natural-key rules.** Custom tables aren't in the hardcoded natural-key list, so `duplicate_natural_key` won't fire on them. That's fine for Phase 4 — Phase 5 can add a UI to register per-table natural keys.

## Acceptance criteria

- [ ] A project whose `discoveredTables` contains `lm_my_custom_table` (with `userClassified = true`, `moduleSlug = "lm"` or in allowed list) runs through extract → cleaning → validate → import without code changes beyond Phase 4.
- [ ] Curated table with the same name as a discovered table still uses the curated definition (no double-extract, no relation conflicts).
- [ ] `npx tsc --noEmit` and `npm run lint` clean.
- [ ] Existing curated-only projects behave identically (no regression).

## Notes for the implementer

- Reuse existing patterns from Phase 2 / Phase 3 — don't introduce new abstractions.
- Curated wins on name collision — document this clearly in the discovery UI ("This table is shadowed by the curated registry and will not be extracted from discovery").
- The hash-to-importOrder function should be stable across runs so the order of extraction is deterministic between extractions of the same project (helps debugging).
